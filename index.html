Module 01

Introduction to Oracle GoldenGate

Oracle GoldenGate is a log-based, real-time data replication and integration platform. Before touching a config file, understand what it does, why it exists, and when to use it.

What Is Oracle GoldenGate?

GoldenGate captures transactional changes from a source database's redo/archive logs and delivers them to one or more targets with sub-second latency — with near-zero impact on the source because it never issues SELECT queries against your tables.

💡
Key insight for junior DBAs GoldenGate reads the redo log, not the tables. This is what gives it near-zero impact on source production databases — one of its biggest advantages over trigger-based or query-based replication tools.

Core Capabilities

  • Real-time CDC — captures INSERT, UPDATE, DELETE from redo logs with sub-second latency
  • Heterogeneous replication — Oracle → MySQL, SQL Server, PostgreSQL, DB2, Kafka, flat files
  • Bidirectional (active-active) — both databases accept writes simultaneously
  • Data transformation — filter rows, remap columns, apply functions during replication
  • Zero-downtime migrations — migrate between platforms, versions, or clouds with no outage
  • Initial load + ongoing sync — bootstrap target then switch to continuous replication seamlessly

Common Use Cases

Use CaseDescriptionTypical Config
Disaster RecoveryHot standby database in sync with production, ready for immediate failoverOracle → Oracle (unidirectional)
Zero-Downtime MigrationMove data between versions, OS, or cloud while source stays onlineOracle 12c → Oracle 19c / Cloud
Real-Time AnalyticsFeed a data warehouse with near-real-time data changesOracle OLTP → Oracle DW / Kafka
Active-Active HATwo or more databases accept writes; changes flow both waysOracle ↔ Oracle (bidirectional)
Data DistributionFan out from one master source to many downstream consumers1 source → multiple targets
Audit / ComplianceCapture all data changes for regulatory compliance or audit trailsOracle → Kafka / flat files

Module 02

GoldenGate Architecture Deep Dive

Understanding every component and how they interact is the foundation of all GoldenGate work. Don't skip this module.

High-Level Data Flow

── SOURCE SERVER ────────────────────────────── TARGET SERVER ── Oracle DB (Redo Logs) │ ▼ EXTRACT Process ──writes──▶ Local Trail (./dirdat/ex000001…) │ │ read by ▼ DATA PUMP ──TCP/IP──▶ COLLECTOR │ writes ▼ Remote Trail (./dirdat/rt000001…) │ ▼ REPLICAT Process │ ▼ Target DB

The Five Core Processes

1. MANAGER — The Supervisor

Manager is the GoldenGate supervisor daemon. It must be running before any other process can start. One Manager per GoldenGate installation.

  • Starts, stops, and monitors all other GoldenGate processes
  • Manages trail file aging (auto-purging old files per PURGEOLDEXTRACTS setting)
  • Listens for incoming connections on TCP port 7809 (default)
  • Auto-restarts failed processes if configured with AUTORESTART

2. EXTRACT — The Capture Engine

Extract reads directly from the database redo logs — never from the tables themselves. This is the key to its near-zero performance impact on the source.

Extract TypeReads FromWrites ToUse When
Primary (Classic)Redo logs directlyLocal trail fileStandard ongoing capture
IntegratedOracle Log Mining ServerLocal trail fileOracle → Oracle — best performance
Data PumpLocal trail fileRemote trail fileNetwork transport and fan-out
Initial LoadDatabase (direct read)Trail or target directlyOne-time bulk sync
✅
Always use Integrated Extract for Oracle-to-Oracle It leverages Oracle's built-in Log Mining Server (LMS), giving better performance, automatic archive log management, and Oracle RAC support.

3. TRAIL FILES — The Message Bus

Trail files are GoldenGate's proprietary binary files that store captured transactions. Think of them as a persistent, ordered message queue between Extract and Replicat.

  • Named: 2-character prefix + 6-digit sequence — e.g. ex000001, ex000002
  • Fixed maximum size (configured in MB); stored under $OGG_HOME/dirdat/
  • Can be inspected using the LOGDUMP utility
  • Can be encrypted with AES-128 or AES-256

4. DATA PUMP — The Network Bridge

A secondary Extract process that reads local trail files and sends data across the network to the target server. Optional but strongly recommended for production.

⭐
Why always use a Data Pump in production Without it, a network outage causes the primary Extract to eventually abend. With a Data Pump, Extract keeps writing to local disk — the Pump catches up automatically when the network recovers. Zero data loss.

5. REPLICAT — The Delivery Engine

Reads remote trail files and applies captured transactions to the target database in correct transaction order.

Replicat TypeParallelismBest For
ClassicSingle-threadedHeterogeneous targets, small volumes
IntegratedMulti-threaded (DB-managed)High-volume Oracle-to-Oracle
ParallelMulti-threaded (OGG-managed)High-volume, non-Oracle target, OGG 12.3+
CoordinatedThread groupsMixed workloads

Supplemental Logging

A critical Oracle database setting that instructs the engine to include extra column data in redo log records. Without it, GoldenGate cannot reconstruct the full before/after row images needed for replication.

-- Check current supplemental logging status:
SELECT SUPPLEMENTAL_LOG_DATA_MIN  AS "MINIMAL",
       SUPPLEMENTAL_LOG_DATA_PK   AS "PK COLS",
       SUPPLEMENTAL_LOG_DATA_ALL  AS "ALL COLS"
FROM   V$DATABASE;

GoldenGate Directory Structure

$OGG_HOME/
├── dirdat/    ← Trail files (local and remote)
├── dirdef/    ← Data definitions files (DEFGEN output)
├── direrr/    ← Discard files (rows rejected by Replicat)
├── dirprm/    ← Parameter files (.prm) — YOUR CONFIG FILES LIVE HERE
├── dirrpt/    ← Process report files (detailed logs per process)
├── dirtmp/    ← Temporary files
├── ggsci      ← GoldenGate CLI tool
├── logdump    ← Trail file inspection utility
└── GLOBALS    ← Global parameter file

Installation & Initial Setup

Step-by-step installation on Linux — database prerequisites, OS setup, and getting the Manager process running.

Database Prerequisites

RequirementCheck CommandExpected
ARCHIVELOG modeSELECT LOG_MODE FROM V$DATABASE;ARCHIVELOG
FORCE LOGGINGSELECT FORCE_LOGGING FROM V$DATABASE;YES
Supplemental loggingSELECT SUPPLEMENTAL_LOG_DATA_MIN FROM V$DATABASE;YES
GGG replication paramSELECT VALUE FROM V$PARAMETER WHERE NAME='enable_goldengate_replication';TRUE

Step-by-Step Installation

1
Enable ARCHIVELOG and database prerequisites
▶
-- Connect as SYSDBA
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE ARCHIVELOG;
ALTER DATABASE OPEN;
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
ALTER DATABASE FORCE LOGGING;
ALTER SYSTEM SET enable_goldengate_replication = TRUE SCOPE=BOTH;
2
Create the GoldenGate database user (both source and target)
▶
CREATE USER ggadmin IDENTIFIED BY Oracle_4U
  DEFAULT TABLESPACE users TEMPORARY TABLESPACE temp QUOTA UNLIMITED ON users;

GRANT CREATE SESSION, ALTER SESSION, SELECT ANY DICTIONARY TO ggadmin;
GRANT SELECT ANY TABLE, FLASHBACK ANY TABLE TO ggadmin;
GRANT EXECUTE ON DBMS_FLASHBACK TO ggadmin;

-- For Integrated Extract (recommended for Oracle source):
EXEC DBMS_GOLDENGATE_AUTH.GRANT_ADMIN_PRIVILEGE('GGADMIN');

-- For Integrated Replicat (target DB):
GRANT DBA TO ggadmin;
3
Create OS user and directories
▶
# As root:
useradd -m -s /bin/bash oracle
mkdir -p /u01/app/oracle/goldengate/19c
chown -R oracle:oinstall /u01/app/oracle/goldengate
4
Extract and install GoldenGate software
▶
# As oracle — extract the downloaded zip:
cd /u01/app/oracle/goldengate/19c
unzip V983788-01.zip
tar -xzf fbo_ggs_Linux_x64_Oracle_shiphome.tar

# Run Oracle Universal Installer:
cd fbo_ggs_Linux_x64_Oracle_shiphome/Disk1
./runInstaller
# GUI steps:
# 1. Select "Oracle GoldenGate for Oracle Database 19c"
# 2. Software Location: /u01/app/oracle/goldengate/19c
# 3. Manager Port: 7809  →  Install
✅
Silent install on headless servers: ./runInstaller -silent -responseFile /tmp/gg.rsp
5
Set environment variables
▶
# Add to /home/oracle/.bash_profile:
export ORACLE_HOME=/u01/app/oracle/product/19c/dbhome_1
export ORACLE_SID=ORCL
export OGG_HOME=/u01/app/oracle/goldengate/19c
export PATH=$OGG_HOME:$ORACLE_HOME/bin:$PATH
export LD_LIBRARY_PATH=$ORACLE_HOME/lib:$OGG_HOME:$LD_LIBRARY_PATH

source ~/.bash_profile
6
Create subdirectories and configure Manager
▶
cd $OGG_HOME && ./ggsci
GGSCI> CREATE SUBDIRS    ← creates dirdat, dirprm, dirrpt, etc.

Create $OGG_HOME/dirprm/mgr.prm:

PORT 7809
DYNAMICPORTLIST 7840-7920
AUTORESTART EXTRACT *, WAITMINUTES 2, RESETMINUTES 5
AUTORESTART REPLICAT *, WAITMINUTES 2, RESETMINUTES 5
PURGEOLDEXTRACTS ./dirdat/*, USECHECKPOINTS, MINKEEPDAYS 3
LAGREPORTHOURS 1
LAGINFOMINUTES 30
LAGCRITICALMINUTES 45
GGSCI> START MANAGER
GGSCI> INFO MANAGER
-- Expected: Manager is running (IP port 7809).
7
Create checkpoint table (target server only)
▶
GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD CHECKPOINTTABLE ggadmin.chkpt
-- Expected: Successfully created checkpoint table ggadmin.chkpt
📌
What is the checkpoint table? It records exactly where Replicat has read in the trail files. If Replicat restarts, it reads this table to resume from the precise position — preventing duplicates or missed transactions.
Module 04

Configuring Unidirectional Replication

Your first end-to-end GoldenGate setup — replicating the HR schema from source to target, step by step, nothing skipped.

Lab Environment

Lab 1 Setup

🖥 Source: sourceserver (SRCDB)
🖥 Target: targetserver (TGTDB)
📁 Schema: HR
🔌 Port: 7809
  • Configure Extract to capture changes from SRCDB
  • Configure Replicat to apply changes to TGTDB
  • Test DML and verify replication within 5 seconds

Step 1 — Enable Supplemental Logging

-- On source DB as SYSDBA:
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;

-- Enable per-table (easier via GGSCI):
GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD TRANDATA hr.employees
GGSCI> ADD TRANDATA hr.departments
GGSCI> ADD TRANDATA hr.jobs
GGSCI> INFO TRANDATA hr.employees

Step 2 — Create Extract Parameter File

Create $OGG_HOME/dirprm/exthr.prm on the source server:

EXTRACT exthr
USERID ggadmin, PASSWORD Oracle_4U
EXTTRAIL ./dirdat/ex
DISCARDFILE ./direrr/exthr.dsc, APPEND, MEGABYTES 100
REPORTCOUNT EVERY 30 MINUTES, RATE

TABLE hr.employees;
TABLE hr.departments;
TABLE hr.jobs;

Step 3 — Register and Start Extract

GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD EXTRACT exthr, INTEGRATED TRANLOG, BEGIN NOW
GGSCI> REGISTER EXTRACT exthr DATABASE
GGSCI> ADD EXTTRAIL ./dirdat/ex, EXTRACT exthr, MEGABYTES 200
GGSCI> START EXTRACT exthr
GGSCI> INFO EXTRACT exthr, DETAIL

Step 4 — Create Replicat Parameter File

Create $OGG_HOME/dirprm/rephr.prm on the target server:

REPLICAT rephr
USERID ggadmin, PASSWORD Oracle_4U
ASSUMETARGETDEFS
DISCARDFILE ./direrr/rephr.dsc, APPEND, MEGABYTES 100
REPORTCOUNT EVERY 30 MINUTES, RATE

REPERROR (DEFAULT, ABEND)
REPERROR (1403, DISCARD)

MAP hr.employees,   TARGET hr.employees;
MAP hr.departments, TARGET hr.departments;
MAP hr.jobs,        TARGET hr.jobs;
📌
ASSUMETARGETDEFS tells Replicat that source and target tables have identical column definitions. If structures differ, use SOURCEDEFS ./dirdef/src.def instead (generated by DEFGEN utility on the source).

Step 5 — Add and Start Replicat

GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD REPLICAT rephr, INTEGRATED, EXTTRAIL ./dirdat/rt, CHECKPOINTTABLE ggadmin.chkpt
GGSCI> START REPLICAT rephr
GGSCI> INFO ALL

Step 6 — Test Replication

-- On SOURCE database:
INSERT INTO hr.employees (employee_id, first_name, last_name, email, hire_date, job_id)
VALUES (999, 'Test', 'User', 'TESTUSER', SYSDATE, 'AD_PRES');
COMMIT;

-- After ~3-5 seconds, on TARGET database:
SELECT employee_id, first_name, last_name
FROM  hr.employees WHERE employee_id = 999;
-- Should return 1 row immediately

-- Monitor stats and lag:
GGSCI> STATS EXTRACT exthr, TOTAL
GGSCI> STATS REPLICAT rephr, TOTAL
GGSCI> LAG EXTRACT exthr
GGSCI> LAG REPLICAT rephr
Module 05

Adding a Data Pump (Production Best Practice)

In Module 04 we sent data directly from Extract to the remote trail. In production, always add a Data Pump between them.

Why You Need a Data Pump in Production

ScenarioWithout Data PumpWith Data Pump
Network outage (1 hour)Extract fills buffer and ABENDs. Data loss possible.Extract writes to local disk. Pump queues. No data loss.
Fan-out to 3 targetsNot possible from primary ExtractOne Data Pump per target — each with different filters
Target maintenanceMust stop Extract (creates gaps)Stop Data Pump only; Extract keeps capturing
Network encryptionPlain-text data on the wireData Pump encrypts before sending

Architecture with Data Pump

SOURCE SERVER: PRIMARY EXTRACT (exthr) │ writes to ▼ Local Trail ./dirdat/ex │ read by ▼ DATA PUMP (pmphr) ──TCP/IP──▶ COLLECTOR │ writes to ▼ Remote Trail ./dirdat/rt │ read by ▼ REPLICAT (rephr)

Step 1 — Update Extract for Local Trail

-- exthr.prm now uses EXTTRAIL (local) instead of RMTTRAIL:
EXTRACT exthr
USERID ggadmin, PASSWORD Oracle_4U
EXTTRAIL ./dirdat/ex
DISCARDFILE ./direrr/exthr.dsc, APPEND, MEGABYTES 100
TABLE hr.*;

Step 2 — Create Data Pump Parameter File

-- $OGG_HOME/dirprm/pmphr.prm (SOURCE SERVER)
EXTRACT pmphr
PASSTHRU
RMTHOST targetserver.example.com, MGRPORT 7809
RMTTRAIL ./dirdat/rt
DISCARDFILE ./direrr/pmphr.dsc, APPEND, MEGABYTES 100
TABLE hr.*;

Step 3 — Add and Start Data Pump

GGSCI> STOP EXTRACT exthr
GGSCI> DELETE RMTTRAIL ./dirdat/rt           -- if previously added
GGSCI> ADD EXTTRAIL ./dirdat/ex, EXTRACT exthr, MEGABYTES 200
GGSCI> START EXTRACT exthr

GGSCI> ADD EXTRACT pmphr, EXTTRAILSOURCE ./dirdat/ex
GGSCI> ADD RMTTRAIL ./dirdat/rt, EXTRACT pmphr, MEGABYTES 200
GGSCI> START EXTRACT pmphr
GGSCI> INFO ALL
Module 06

Initial Data Load (Instantiation)

Before ongoing replication starts, the target must be populated with a consistent snapshot of the source. How you do this determines whether you get gaps or duplicates.

The SCN-Based Approach (Zero Downtime)

Timeline: T1: Record current SCN = 4892031 T2: Take EXPDP snapshot at SCN 4892031 (source stays online) T3: Extract starts queueing changes AFTER SCN 4892031 T4: Import snapshot to target (may take minutes/hours) T5: Start Replicat — applies all queued changes since 4892031 T6: Lag reaches near-zero — replication is live! No downtime. No gaps. No duplicates.

Step 1 — Get Snapshot SCN

SELECT CURRENT_SCN FROM V$DATABASE;
-- Example: 4892031  ← WRITE THIS DOWN

Step 2 — Register Extract at That SCN (don't start yet)

GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD EXTRACT exthr, INTEGRATED TRANLOG, SCN 4892031
GGSCI> REGISTER EXTRACT exthr DATABASE
GGSCI> ADD EXTTRAIL ./dirdat/ex, EXTRACT exthr, MEGABYTES 200
-- DO NOT start Extract yet!

Step 3 — Export Source Data at That SCN

expdp ggadmin/Oracle_4U \
  SCHEMAS=hr \
  DIRECTORY=DATA_PUMP_DIR \
  DUMPFILE=hr_initial.dmp \
  LOGFILE=hr_export.log \
  FLASHBACK_SCN=4892031

Step 4 — Import to Target

impdp ggadmin/Oracle_4U \
  SCHEMAS=hr \
  DIRECTORY=DATA_PUMP_DIR \
  DUMPFILE=hr_initial.dmp \
  LOGFILE=hr_import.log \
  REMAP_SCHEMA=hr:hr \
  TABLE_EXISTS_ACTION=REPLACE

Step 5 — Start All Processes

GGSCI> START EXTRACT exthr
GGSCI> START EXTRACT pmphr
GGSCI> START REPLICAT rephr

-- Monitor lag dropping toward zero:
GGSCI> LAG REPLICAT rephr
⚠️
HANDLECOLLISIONS during sync While Replicat catches up you may get duplicate-key errors. Add HANDLECOLLISIONS to rephr.prm temporarily. Remove it once lag is near-zero and the run is clean.
Module 07

DDL Replication

By default GoldenGate only replicates DML. DDL replication adds support for CREATE TABLE, ALTER TABLE, DROP TABLE, and other schema changes.

📌
When you need DDL replication In long-running environments, application deployments add columns, create tables, or modify indexes. Without DDL replication these changes must be manually applied to the target — which is error-prone and causes replication to break.

Install DDL Support Objects (Oracle Source)

# Run as SYSDBA on source:
cd $OGG_HOME
sqlplus / as sysdba

SQL> @marker_setup.sql
SQL> @ddl_setup.sql
SQL> @role_setup.sql
SQL> GRANT GGS_GGSUSER_ROLE TO ggadmin;
SQL> @ddl_enable.sql
SQL> @ddl_pin.sql

Configure DDL in Extract

EXTRACT exthr
USERID ggadmin, PASSWORD Oracle_4U
EXTTRAIL ./dirdat/ex

DDL INCLUDE MAPPED      ← only DDL for tables in TABLE statements
DDLOPTIONS REPORT

TABLE hr.employees;
TABLE hr.departments;

Configure DDL in Replicat

REPLICAT rephr
USERID ggadmin, PASSWORD Oracle_4U
ASSUMETARGETDEFS

DDL INCLUDE MAPPED
DDLERROR DEFAULT ABEND RETRYOP MAXRETRIES 3 RETRYDELAY 5

MAP hr.employees,   TARGET hr.employees;
MAP hr.departments, TARGET hr.departments;

DDL Include Options

OptionWhat It ReplicatesRecommendation
DDL INCLUDE MAPPEDOnly DDL for tables in TABLE/MAP statements✅ Safest — use this
DDL INCLUDE ALLAll DDL in the database⚠️ Risky — includes system DDL
DDL INCLUDE UNMAPPEDDDL for objects NOT in TABLE/MAP❌ Rarely needed
Module 08

Data Filtering & Transformation

GoldenGate can filter rows, remap columns, rename tables, and transform values as data flows through the pipeline.

Row Filtering with WHERE

-- Replicate only active employees:
MAP hr.employees, TARGET hr.employees
  WHERE (status = 'ACTIVE');

-- Filter in Extract (better — reduces trail file size):
TABLE hr.employees, WHERE (department_id IN (10, 20, 30));

Column Mapping & Transformation

-- Map to differently-named target columns + compute values:
MAP hr.employees, TARGET dw.dim_employees
  COLMAP (emp_id    = employee_id,
          full_name = @UPPER(last_name) || ', ' || first_name,
          dept_id   = department_id,
          load_ts   = @DATENOW());

-- Conditional transformation:
MAP hr.orders, TARGET dw.fact_orders
  COLMAP (order_id   = order_id,
          amount_usd = @COMPUTE(order_amount * exchange_rate),
          order_dt   = @DATE('YYYY-MM-DD', 'DD-MON-YYYY', order_date),
          status_lbl = @IF(status='A', 'Active', 'Inactive'));

Built-in Conversion Functions

FunctionPurposeExample
@COMPUTEArithmetic operations@COMPUTE(price * qty)
@DATEDate format conversion@DATE('YYYY-MM-DD','DD-MON-YY',dt)
@DATENOWCurrent timestamp@DATENOW()
@UPPER / @LOWERString case@UPPER(last_name)
@SUBSTRSubstring extraction@SUBSTR(name,1,20)
@IFConditional value@IF(status='A','Active','Inactive')
@GETENVEnvironment/header variable@GETENV('SYSTEM','HOSTNAME')
@TOKENRead a user-defined token@TOKEN('TK-HOST')

Token-Based Data Enrichment

-- In Extract — attach metadata tokens to each captured row:
TABLE hr.employees,
  TOKENS (TK-HOST   = @GETENV('SYSTEM', 'HOSTNAME'),
          TK-TIME   = @DATENOW(),
          TK-OPTYPE = @GETENV('GGHEADER', 'OPTYPE'));

-- In Replicat — use token values to populate an audit table:
MAP hr.employees, TARGET hr.emp_audit
  COLMAP (employee_id = employee_id,
          last_name   = last_name,
          src_host    = @TOKEN('TK-HOST'),
          change_time = @TOKEN('TK-TIME'),
          op_type     = @TOKEN('TK-OPTYPE'));

Handling ABEND Errors

-- Step 1: Find the error:
GGSCI> VIEW REPORT rephr   ← scroll to the bottom

-- Option A: Skip the bad transaction (last resort!):
GGSCI> START REPLICAT rephr, SKIPTRANSACTION

-- Option B: Skip to a known-good SCN:
GGSCI> START REPLICAT rephr, AFTERCSN 4892099

-- Option C: Handle collisions temporarily:
-- Add HANDLECOLLISIONS to rephr.prm, then restart normally

-- Option D: Fix the data issue, then:
GGSCI> START REPLICAT rephr
⚠️
SKIPTRANSACTION discards an entire transaction. Always understand the root cause first. Log skipped transactions by ensuring DISCARDFILE is in your parameter file.
Module 10

Bidirectional (Active-Active) Replication

Both databases accept writes simultaneously and changes flow in both directions. The most powerful — and most complex — GoldenGate configuration.

DATABASE A (Primary DC) DATABASE B (Secondary DC) Extract A (extA) ───────────────────▶ Replicat B (repB) Replicat A (repA) ◀─────────────────── Extract B (extB) Both databases: READ + WRITE (active-active) Loop prevention: CRITICAL — without it, changes loop forever

Loop Prevention

-- On Database A's Extract (extA.prm):
EXTRACT extA
TRANLOGOPTIONS EXCLUDETAG 00    ← exclude changes tagged '00' (Replicat's tag)
EXTTRAIL ./dirdat/ea
TABLE hr.*;
✅
How it works: When Replicat applies a transaction, Oracle writes a marker (tag 00) in the redo log. The Extract on that same DB reads the redo log and skips anything tagged 00 — because that came from the other database, not a local application write.

Conflict Detection & Resolution (CDR)

-- In repA.prm — most-recent timestamp wins:
MAP hr.employees, TARGET hr.employees,
  RESOLVECONFLICT (UPDATEROWEXISTS,
    (DEFAULT, USEMAX (last_updated)));

-- Ignore delete conflicts:
MAP hr.orders, TARGET hr.orders,
  RESOLVECONFLICT (DELETEROWEXISTS, (DEFAULT, DISCARD));

-- Use numeric delta for counters:
MAP hr.inventory, TARGET hr.inventory,
  RESOLVECONFLICT (UPDATEROWEXISTS,
    (DEFAULT, USEDELTA (quantity_on_hand)));

CDR Resolution Strategies

StrategyDescriptionBest For
USEMAXKeep the larger value of the conflicting columnTimestamps (most-recent-wins)
USEMINKeep the smaller valuePriority queues, minimum bids
USEDELTAApply the numeric difference from sourceAccount balances, inventory counts
OVERWRITESource always overwrites targetHA failover with a designated master
DISCARDIgnore the conflict, keep target valueDelete-row-not-found conflicts

DECRYPTTRAIL AES256

Troubleshooting Guide

A systematic approach to diagnosing and resolving GoldenGate issues — from minor ABENDs to major re-sync scenarios.

Troubleshooting Framework — Always Follow This Order

  1. Run INFO ALL — identify which process is ABENDED or STOPPED
  2. Run VIEW REPORT <processname> — the error is at the bottom of the file
  3. Run VIEW GGSEVT — check for system-level GoldenGate events
  4. Check OS logs: /var/log/messages, Oracle alert log
  5. Use LOGDUMP to inspect trail file contents if needed
  6. Check database views: V$GOLDENGATE_STATS, alert log

Common Errors & Resolutions

ErrorRoot CauseResolution
OGG-00446 Error opening log fileArchive log deleted before Extract captured itIncrease archive retention; check ARCHIVELOG_DEST
OGG-01161 Bad column lengthColumn definition mismatch source vs. targetRun DEFGEN; update SOURCEDEFS; check column sizes
ORA-01403 No data foundDELETE/UPDATE for row that doesn't exist on targetAdd REPERROR(1403, DISCARD); verify initial load
ORA-00001 Unique constraintDuplicate key — initial load + Replicat overlapUse HANDLECOLLISIONS during sync; remove after
OGG-01224 TCP/IP errorNetwork to target Manager failedCheck firewall; verify Manager running; verify port 7809
OGG-00664 OCI error fetching rowLong-running uncommitted transactionFind and commit/rollback the long transaction on source
Missing archived logArchives purged before Extract processed themIncrease archive retention policy

Emergency Re-Synchronization Procedure

  1. Stop all GoldenGate processes immediately
  2. Document the situation: what trail files exist, last known good Replicat checkpoint
  3. If trail files cover the gap → reset Replicat to last good checkpoint and restart
  4. If trail files are missing → perform a new initial load (Module 06)
  5. After re-sync: start with HANDLECOLLISIONS, monitor 24h, then remove it
-- Check Replicat's last checkpoint:
SELECT group_name, log_csn, log_position
FROM   ggadmin.chkpt
ORDER BY last_update_ts DESC
FETCH FIRST 5 ROWS ONLY;

Database Health Check SQL

-- Run on source DB to verify all GoldenGate prerequisites:
SELECT 'DB Mode' chk, LOG_MODE || ' | FORCE_LOG: ' || FORCE_LOGGING val FROM V$DATABASE
UNION ALL SELECT 'GGG Param', VALUE FROM V$PARAMETER WHERE NAME='enable_goldengate_replication'
UNION ALL SELECT 'Suppl Log',
  'MIN:'||SUPPLEMENTAL_LOG_DATA_MIN||' PK:'||SUPPLEMENTAL_LOG_DATA_PK||' ALL:'||SUPPLEMENTAL_LOG_DATA_ALL
  FROM V$DATABASE;

-- Find long-running uncommitted transactions (cause of Extract lag):
SELECT s.username, t.start_time, t.used_ublk, s.status
FROM  v$transaction t JOIN v$session s ON t.ses_addr = s.saddr
ORDER BY t.start_time;
