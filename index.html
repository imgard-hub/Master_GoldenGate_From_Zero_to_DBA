Module 01

Introduction to Oracle GoldenGate

Oracle GoldenGate is a comprehensive software package for real-time data integration and replication in heterogeneous IT environments. Before touching a single configuration file, understand what it does, why it exists, and when to use it.

What Is Oracle GoldenGate?

GoldenGate is a log-based, real-time data replication and integration platform. It captures transactional changes from a source database's redo/archive logs and delivers them to one or more target systems with sub-second latency — all with minimal impact on the source database because it never issues SELECT queries against your tables.

💡
Key insight for junior DBAs GoldenGate reads the database transaction log (redo log), not the actual tables. This means it has near-zero performance impact on source production databases — one of its biggest advantages over trigger-based or query-based replication.

Core Capabilities

  • Real-time CDC — Captures inserts, updates, deletes from redo logs with sub-second latency
  • Heterogeneous replication — Oracle to MySQL, SQL Server, PostgreSQL, DB2, Kafka, flat files, and more
  • Bidirectional (active-active) — Both databases accept writes simultaneously for high availability
  • Data transformation — Filter rows, remap columns, apply functions during replication
  • Zero-downtime migrations — Migrate between platforms, versions, or clouds with no outage
  • Initial load + ongoing sync — Bootstrap the target then switch to continuous replication seamlessly

Common Use Cases

Use CaseDescriptionTypical Config
Disaster RecoveryMaintain a hot standby database in sync with production, ready for immediate failoverOracle → Oracle (unidirectional)
Zero-Downtime MigrationMove data between versions, OS, or cloud platforms while the source stays onlineOracle 12c → Oracle 19c / Cloud
Real-Time AnalyticsFeed a data warehouse or reporting system with near-real-time data changesOracle OLTP → Oracle DW / Kafka
Active-Active HATwo or more databases accept writes; changes flow in both directionsOracle ↔ Oracle (bidirectional)
Data DistributionFan out data from one master source to many downstream consumers1 source → multiple targets
Audit / ComplianceCapture all data changes for regulatory compliance or audit trailsOracle → Kafka / flat files

Module 02

GoldenGate Architecture Deep Dive

Understanding every component, how they interact, and why they're designed that way is the foundation of all GoldenGate work. Don't skip this module.

High-Level Data Flow

──── SOURCE SERVER ─────────────────────────────────────────────── TARGET SERVER ──── Oracle DB │ (Redo / Archive Logs) ▼ EXTRACT Process ──writes──▶ Local Trail (./dirdat/ex000001 …) │ │ read by ▼ DATA PUMP ──TCP/IP──▶ COLLECTOR │ writes ▼ Remote Trail (./dirdat/rt000001 …) │ │ read by ▼ REPLICAT Process │ ▼ Target DB

The Five Core Processes

1. MANAGER — The Supervisor

Manager is the GoldenGate supervisor daemon. It must be running before any other process can start. There is exactly one Manager per GoldenGate installation.

  • Starts, stops, and monitors all other GoldenGate processes
  • Manages trail file aging (auto-purging old trail files)
  • Listens for incoming connections on TCP port 7809 (default)
  • Auto-restarts failed processes (if configured with AUTORESTART)
  • Sends threshold alerts when lag is too high

2. EXTRACT — The Capture Engine

Extract reads directly from the database redo logs — never from the tables themselves. This is the key to its near-zero performance impact.

Extract TypeReads FromWrites ToUse When
Primary (Classic)Redo logs directlyLocal trail fileStandard ongoing capture
IntegratedOracle Log Mining ServerLocal trail fileOracle → Oracle, best performance
Data PumpLocal trail fileRemote trail fileNetwork transport, fan-out
Initial LoadDatabase (direct read)Trail or target directlyOne-time bulk sync before replication starts
✅
Always use Integrated Extract for Oracle-to-Oracle Integrated Extract leverages Oracle's built-in Log Mining Server (LMS), which gives better performance, automatic archive log management, and support for Oracle RAC environments.

3. TRAIL FILES — The Message Bus

Trail files are GoldenGate's proprietary binary files that store captured transactions. Think of them as a persistent, ordered message queue between Extract and Replicat.

  • Named with a 2-character prefix + 6-digit sequence: ex000001, ex000002, …
  • Each file is a fixed maximum size (configured in MB, default 500 MB)
  • Stored under $OGG_HOME/dirdat/ by default
  • Read by the LOGDUMP utility for debugging
  • Can be encrypted with AES-128 or AES-256
Trail TypeLocationWritten ByRead By
Local TrailSource serverPrimary ExtractData Pump Extract
Remote TrailTarget serverData Pump via CollectorReplicat

4. DATA PUMP — The Network Bridge

The Data Pump is a secondary Extract process that reads local trail files and sends the data across the network to the target server's trail files. It is optional but strongly recommended for production.

⭐
Why always use a Data Pump in production If there's no Data Pump and the network goes down, the primary Extract has nowhere to write and will eventually abend. With a Data Pump, the primary Extract keeps writing to local trail files on disk. The Data Pump catches up automatically when the network recovers. Zero data loss.

5. REPLICAT — The Delivery Engine

Replicat reads remote trail files and applies the captured transactions to the target database, reconstructing the original DML and DDL operations in transaction order.

Replicat TypeParallelismBest ForOracle Version
ClassicSingle-threadedHeterogeneous targets, small volumesAll versions
IntegratedMulti-threaded (DB-managed)High-volume Oracle-to-OracleOracle 11.2.0.3+
ParallelMulti-threaded (OGG-managed)High-volume, non-Oracle targetOGG 12.3+
CoordinatedThread groupsMixed workloadsOGG 12.1+

Supplemental Logging

Supplemental logging is a critical Oracle database setting that instructs the database engine to include extra column data in redo log records. Without it, GoldenGate cannot reconstruct the full before/after row images needed for replication.

By default, Oracle only logs the minimum information needed for database recovery — not enough for GoldenGate. Supplemental logging adds the column values that GoldenGate requires.

  • Minimal (database-level) — Required minimum; enables redo to identify changed rows
  • PRIMARY KEY — Logs PK columns on every UPDATE/DELETE (lower redo overhead)
  • ALL COLUMNS — Logs every column on every change (highest fidelity, more redo)
-- Verify current supplemental logging status
SELECT SUPPLEMENTAL_LOG_DATA_MIN   AS "MINIMAL",
       SUPPLEMENTAL_LOG_DATA_PK    AS "PK COLS",
       SUPPLEMENTAL_LOG_DATA_ALL   AS "ALL COLS"
FROM   V$DATABASE;

Installation & Initial Setup

A step-by-step guide to installing Oracle GoldenGate on Linux, configuring the database prerequisites, and getting the Manager process running.

Pre-Installation: Database Requirements

RequirementCommand to CheckExpected Value
ARCHIVELOG modeSELECT LOG_MODE FROM V$DATABASE;ARCHIVELOG
FORCE LOGGINGSELECT FORCE_LOGGING FROM V$DATABASE;YES
Supplemental loggingSELECT SUPPLEMENTAL_LOG_DATA_MIN FROM V$DATABASE;YES
GGG replication paramSELECT VALUE FROM V$PARAMETER WHERE NAME='enable_goldengate_replication';TRUE

Step-by-Step Installation

1
Enable ARCHIVELOG mode and database prerequisites
▶
-- Connect as SYSDBA
SHUTDOWN IMMEDIATE;
STARTUP MOUNT;
ALTER DATABASE ARCHIVELOG;
ALTER DATABASE OPEN;
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;
ALTER DATABASE FORCE LOGGING;
ALTER SYSTEM SET enable_goldengate_replication = TRUE SCOPE=BOTH;
2
Create the GoldenGate database user (both source and target)
▶
CREATE USER ggadmin IDENTIFIED BY Oracle_4U
  DEFAULT TABLESPACE users
  TEMPORARY TABLESPACE temp
  QUOTA UNLIMITED ON users;

GRANT CREATE SESSION, ALTER SESSION TO ggadmin;
GRANT SELECT ANY DICTIONARY TO ggadmin;
GRANT SELECT ANY TABLE TO ggadmin;
GRANT FLASHBACK ANY TABLE TO ggadmin;
GRANT EXECUTE ON DBMS_FLASHBACK TO ggadmin;

-- For Integrated Extract (recommended):
EXEC DBMS_GOLDENGATE_AUTH.GRANT_ADMIN_PRIVILEGE('GGADMIN');

-- For Integrated Replicat (target DB):
GRANT DBA TO ggadmin;
3
Create OS user and GoldenGate home directory
▶
# As root:
useradd -m -s /bin/bash oracle
mkdir -p /u01/app/oracle/goldengate/19c
chown -R oracle:oinstall /u01/app/oracle/goldengate
4
Extract and install GoldenGate software
▶
# As oracle user — extract the downloaded zip:
cd /u01/app/oracle/goldengate/19c
unzip V983788-01.zip
tar -xzf fbo_ggs_Linux_x64_Oracle_shiphome.tar

# Run the Oracle Universal Installer:
cd fbo_ggs_Linux_x64_Oracle_shiphome/Disk1
./runInstaller
# In the GUI:
# 1. Select "Oracle GoldenGate for Oracle Database 19c"
# 2. Software Location: /u01/app/oracle/goldengate/19c
# 3. Manager Port: 7809
# 4. Click Install
✅
Silent install For headless servers: ./runInstaller -silent -responseFile /tmp/gg_install.rsp
5
Set environment variables
▶
# Add to /home/oracle/.bash_profile:
export ORACLE_HOME=/u01/app/oracle/product/19c/dbhome_1
export ORACLE_SID=ORCL
export OGG_HOME=/u01/app/oracle/goldengate/19c
export PATH=$OGG_HOME:$ORACLE_HOME/bin:$PATH
export LD_LIBRARY_PATH=$ORACLE_HOME/lib:$OGG_HOME:$LD_LIBRARY_PATH

# Reload:
source ~/.bash_profile
6
Create subdirectories and configure Manager
▶
# Start GGSCI and create all required directories:
cd $OGG_HOME && ./ggsci
GGSCI> CREATE SUBDIRS

Now create the Manager parameter file at $OGG_HOME/dirprm/mgr.prm:

-- $OGG_HOME/dirprm/mgr.prm
PORT 7809
DYNAMICPORTLIST 7840-7920
AUTORESTART EXTRACT *, WAITMINUTES 2, RESETMINUTES 5
AUTORESTART REPLICAT *, WAITMINUTES 2, RESETMINUTES 5
PURGEOLDEXTRACTS ./dirdat/*, USECHECKPOINTS, MINKEEPDAYS 3
LAGREPORTHOURS 1
LAGINFOMINUTES 30
LAGCRITICALMINUTES 45
# Start Manager:
GGSCI> START MANAGER
GGSCI> INFO MANAGER
# Expected: Manager is running (IP port 7809).
7
Create checkpoint table (target server only)
▶
# On TARGET server in GGSCI:
GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD CHECKPOINTTABLE ggadmin.chkpt
# Expected: Successfully created checkpoint table ggadmin.chkpt
📌
What is the checkpoint table? It records where Replicat has read up to in the trail files. If Replicat restarts, it reads this table to resume from the exact right position — preventing duplicate or missed transactions.
Module 04

Configuring Unidirectional Replication

Your first end-to-end GoldenGate setup. We'll replicate the HR schema from a source Oracle database to a target Oracle database — step by step, no steps skipped.

Lab Environment

Lab 1 Setup

🖥️ Source: sourceserver (SRCDB)
🖥️ Target: targetserver (TGTDB)
📁 Schema: HR (employees, departments, jobs)
🔌 Port: 7809 (both servers)
  • Configure Extract to capture changes from SRCDB
  • Configure Replicat to apply changes to TGTDB
  • Test with live DML and verify replication within 5 seconds

Source-Side: Enable Supplemental Logging

-- On source DB (SRCDB) as SYSDBA:
ALTER DATABASE ADD SUPPLEMENTAL LOG DATA;

-- Enable for each table (use GGSCI ADD TRANDATA — easier):
GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U
GGSCI> ADD TRANDATA hr.employees
GGSCI> ADD TRANDATA hr.departments
GGSCI> ADD TRANDATA hr.jobs

-- Verify:
GGSCI> INFO TRANDATA hr.employees
-- Expected: Logging of supplemental redo log data is enabled for table HR.EMPLOYEES.

Source-Side: Create Extract Parameter File

Create the file $OGG_HOME/dirprm/exthr.prm on the source server:

-- $OGG_HOME/dirprm/exthr.prm (SOURCE SERVER)
EXTRACT exthr
USERID ggadmin, PASSWORD Oracle_4U
EXTTRAIL ./dirdat/ex
DISCARDFILE ./direrr/exthr.dsc, APPEND, MEGABYTES 100
REPORTCOUNT EVERY 30 MINUTES, RATE

-- Tables to capture:
TABLE hr.employees;
TABLE hr.departments;
TABLE hr.jobs;

Source-Side: Register and Start Extract

GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U

-- Register Extract (BEGIN NOW = start capturing from this moment forward):
GGSCI> ADD EXTRACT exthr, INTEGRATED TRANLOG, BEGIN NOW
GGSCI> REGISTER EXTRACT exthr DATABASE

-- Add the local trail file:
GGSCI> ADD EXTTRAIL ./dirdat/ex, EXTRACT exthr, MEGABYTES 200

GGSCI> START EXTRACT exthr

-- Verify it's running:
GGSCI> INFO EXTRACT exthr, DETAIL

Target-Side: Create Replicat Parameter File

Create $OGG_HOME/dirprm/rephr.prm on the target server:

-- $OGG_HOME/dirprm/rephr.prm (TARGET SERVER)
REPLICAT rephr
USERID ggadmin, PASSWORD Oracle_4U
ASSUMETARGETDEFS
DISCARDFILE ./direrr/rephr.dsc, APPEND, MEGABYTES 100
REPORTCOUNT EVERY 30 MINUTES, RATE

-- Error handling:
REPERROR (DEFAULT, ABEND)
REPERROR (1403, DISCARD)   -- ORA-01403: row not found on DELETE = ignore

-- Table mappings:
MAP hr.employees,  TARGET hr.employees;
MAP hr.departments, TARGET hr.departments;
MAP hr.jobs,       TARGET hr.jobs;
📌
ASSUMETARGETDEFS tells Replicat that source and target tables have identical column definitions. If your structures differ, use SOURCEDEFS ./dirdef/src.def instead (generated by DEFGEN utility on the source).

Target-Side: Add and Start Replicat

GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U

-- Add Replicat pointing at the remote trail:
GGSCI> ADD REPLICAT rephr, INTEGRATED, EXTTRAIL ./dirdat/rt, CHECKPOINTTABLE ggadmin.chkpt

GGSCI> START REPLICAT rephr

-- Verify:
GGSCI> INFO REPLICAT rephr, DETAIL
GGSCI> INFO ALL

Testing the Replication

-- On SOURCE database — insert a test row:
INSERT INTO hr.employees (employee_id, first_name, last_name, email, hire_date, job_id)
VALUES (999, 'Test', 'User', 'TESTUSER', SYSDATE, 'AD_PRES');
COMMIT;

-- Wait ~3-5 seconds, then on TARGET database:
SELECT employee_id, first_name, last_name
FROM  hr.employees
WHERE employee_id = 999;
-- Should return 1 row immediately
-- Monitor statistics (run on either server):
GGSCI> STATS EXTRACT exthr, TOTAL
GGSCI> STATS REPLICAT rephr, TOTAL
GGSCI> LAG EXTRACT exthr
GGSCI> LAG REPLICAT rephr
✅
What good lag looks like Under normal load: Extract lag 0-2 seconds, Replicat lag 2-5 seconds. Total end-to-end latency under 10 seconds is excellent.
Module 05

Adding a Data Pump (Production Best Practice)

In Module 4 we sent data directly from Extract to the remote trail. In production, always add a Data Pump between them. Here's why — and how.

Why You Need a Data Pump in Production

ScenarioWithout Data PumpWith Data Pump
Network outage (1 hour)Extract fills its buffer and ABENDs. Data loss possible.Extract writes to local disk. Data Pump queues. No data loss.
Fan-out to 3 targetsNot possible from primary ExtractOne Data Pump per target — each can filter/transform differently
Target server maintenanceMust stop Extract (gaps data)Stop Data Pump only; Extract keeps capturing
Network encryptionPlain-text trail data on the wireData Pump encrypts trail before sending

Architecture with Data Pump

SOURCE SERVER: PRIMARY EXTRACT (exthr) │ writes to ▼ Local Trail ./dirdat/ex │ read by ▼ DATA PUMP (pmphr) ──── TCP/IP ────▶ COLLECTOR (target, dynamic) │ writes to ▼ Remote Trail ./dirdat/rt │ read by ▼ REPLICAT (rephr)

Step 1 — Reconfigure Extract to Write Local Trail

Modify exthr.prm to use EXTTRAIL (local) instead of RMTTRAIL (remote):

-- $OGG_HOME/dirprm/exthr.prm — UPDATED
EXTRACT exthr
USERID ggadmin, PASSWORD Oracle_4U
EXTTRAIL ./dirdat/ex          ← LOCAL trail now
DISCARDFILE ./direrr/exthr.dsc, APPEND, MEGABYTES 100

TABLE hr.employees;
TABLE hr.departments;
TABLE hr.jobs;

Step 2 — Create Data Pump Parameter File

-- $OGG_HOME/dirprm/pmphr.prm (SOURCE SERVER)
EXTRACT pmphr
PASSTHRU                                  ← No processing, just forward
RMTHOST targetserver.example.com, MGRPORT 7809
RMTTRAIL ./dirdat/rt
DISCARDFILE ./direrr/pmphr.dsc, APPEND, MEGABYTES 100

TABLE hr.*;

Step 3 — Add and Start Data Pump

-- Stop Extract, update trail, restart:
GGSCI> STOP EXTRACT exthr
GGSCI> DELETE RMTTRAIL ./dirdat/rt
GGSCI> ADD EXTTRAIL ./dirdat/ex, EXTRACT exthr, MEGABYTES 200
GGSCI> START EXTRACT exthr

-- Add Data Pump (reads from local trail):
GGSCI> ADD EXTRACT pmphr, EXTTRAILSOURCE ./dirdat/ex
GGSCI> ADD RMTTRAIL ./dirdat/rt, EXTRACT pmphr, MEGABYTES 200
GGSCI> START EXTRACT pmphr

-- Check everything is running:
GGSCI> INFO ALL
📌
PASSTHRU means the Data Pump forwards data unchanged. If you need to apply filtering or transformations at the pump level (e.g., different rules per target), replace PASSTHRU with TABLE and MAP statements.
Module 06

Initial Data Load (Instantiation)

Before ongoing replication can start, the target database must be populated with a consistent snapshot of the source. This is called the "initial load" or "instantiation" — and how you do it determines whether you have data gaps or duplicates.

The SCN-Based Approach (Zero Downtime)

The key insight: use Oracle's System Change Number (SCN) to coordinate the snapshot and the start of ongoing replication. GoldenGate will queue all changes from the snapshot SCN while you import the data — ensuring zero gaps and zero duplicates.

Timeline: T1: Record current SCN = 4892031 T2: Take EXPDP snapshot at SCN 4892031 (source stays fully online) T3: Extract starts queueing changes AFTER SCN 4892031 T4: Import snapshot to target (may take minutes/hours) T5: Start Replicat — applies all queued changes since 4892031 T6: Lag reaches near-zero — replication is live! No downtime. No gaps. No duplicates.

Step 1 — Get the Snapshot SCN

-- On SOURCE database:
SELECT CURRENT_SCN FROM V$DATABASE;
-- Example output: 4892031   ← Write this down!

Step 2 — Register Extract at That SCN (Don't Start It Yet)

GGSCI> DBLOGIN USERID ggadmin PASSWORD Oracle_4U

-- Register at the snapshot SCN:
GGSCI> ADD EXTRACT exthr, INTEGRATED TRANLOG, SCN 4892031
GGSCI> REGISTER EXTRACT exthr DATABASE
GGSCI> ADD EXTTRAIL ./dirdat/ex, EXTRACT exthr, MEGABYTES 200

-- DO NOT start Extract yet!

Step 3 — Export Source Data at That SCN

expdp ggadmin/Oracle_4U \
  SCHEMAS=hr \
  DIRECTORY=DATA_PUMP_DIR \
  DUMPFILE=hr_initial.dmp \
  LOGFILE=hr_export.log \
  FLASHBACK_SCN=4892031

Step 4 — Import to Target Database

impdp ggadmin/Oracle_4U \
  SCHEMAS=hr \
  DIRECTORY=DATA_PUMP_DIR \
  DUMPFILE=hr_initial.dmp \
  LOGFILE=hr_import.log \
  REMAP_SCHEMA=hr:hr \
  TABLE_EXISTS_ACTION=REPLACE

Step 5 — Start Extract, Data Pump, and Replicat

-- Source: start Extract (it queued changes since SCN 4892031):
GGSCI> START EXTRACT exthr
GGSCI> START EXTRACT pmphr

-- Target: start Replicat:
GGSCI> START REPLICAT rephr

-- Monitor lag — watch it drop toward zero:
GGSCI> LAG REPLICAT rephr
⚠️
HANDLECOLLISIONS during sync While Replicat is catching up, you may get duplicate-key errors if some rows from the import overlap with rows in the trail. Add HANDLECOLLISIONS to rephr.prm temporarily. Remove it once lag is near-zero and verify a clean run.

Initial Load Methods Comparison

MethodBest ForDowntimeComplexity
EXPDP/IMPDP + SCN (above)Oracle-to-Oracle, large schemasZeroMedium
GoldenGate Direct LoadSmall tables, heterogeneous targetsZeroLow
RMAN restoreVery large databases (>1 TB)MinimalMedium
File-based bulk loadNon-Oracle targetsVariesHigh
Module 07

DDL Replication

By default, GoldenGate only replicates DML (INSERT, UPDATE, DELETE). DDL replication adds support for schema changes like CREATE TABLE, ALTER TABLE, and DROP TABLE.

📌
When you need DDL replication In long-running replication environments, application deployments add new columns, create new tables, or modify indexes. Without DDL replication, these schema changes must be manually applied to the target — which is error-prone and causes replication to break.

Install DDL Support Objects (Oracle Source)

# Run as SYSDBA on source database:
cd $OGG_HOME
sqlplus / as sysdba

SQL> @marker_setup.sql
SQL> @ddl_setup.sql
SQL> @role_setup.sql
SQL> GRANT GGS_GGSUSER_ROLE TO ggadmin;
SQL> @ddl_enable.sql
SQL> @ddl_pin.sql

Configure DDL in Extract (exthr.prm)

EXTRACT exthr
USERID ggadmin, PASSWORD Oracle_4U
EXTTRAIL ./dirdat/ex

-- DDL capture (only for mapped objects — safest):
DDL INCLUDE MAPPED
DDLOPTIONS REPORT             ← Log all DDL to report file

TABLE hr.employees;
TABLE hr.departments;

Configure DDL in Replicat (rephr.prm)

REPLICAT rephr
USERID ggadmin, PASSWORD Oracle_4U
ASSUMETARGETDEFS

DDL INCLUDE MAPPED
DDLERROR DEFAULT ABEND RETRYOP MAXRETRIES 3 RETRYDELAY 5

MAP hr.employees,  TARGET hr.employees;
MAP hr.departments, TARGET hr.departments;

DDL Include Options Reference

OptionWhat It ReplicatesRecommended
DDL INCLUDE MAPPEDOnly DDL for tables in TABLE/MAP statements✅ Yes — safest
DDL INCLUDE ALLAll DDL in the database⚠️ Risky — includes system DDL
DDL INCLUDE UNMAPPEDDDL for objects NOT in TABLE/MAP❌ Rarely needed
DDL EXCLUDEExclude DDL entirelyDefault if no DDL clause present
Module 08

Data Filtering & Transformation

GoldenGate can filter rows, remap columns, rename tables, and transform values as data flows through the pipeline. These can be applied in Extract, Data Pump, or Replicat.

Row Filtering with WHERE

-- Replicate only employees in departments 10, 20, or 30:
MAP hr.employees, TARGET hr.employees
  WHERE (department_id IN (10, 20, 30));

-- Filter in Extract (better: reduces trail file size):
TABLE hr.employees, WHERE (department_id IN (10, 20, 30));

-- Exclude cancelled orders:
TABLE hr.orders, WHERE (status <> 'CANCELLED');

Column Mapping & Renaming

-- Map source columns to differently-named target columns:
MAP hr.employees, TARGET dw.dim_employees
  COLMAP (emp_id    = employee_id,
          full_name = UPPER(last_name) || ', ' || first_name,
          dept_id   = department_id,
          load_ts   = @DATENOW());       ← Adds a timestamp column

-- Computed column transformations:
MAP hr.orders, TARGET dw.fact_orders
  COLMAP (order_id   = order_id,
          amount_usd = @COMPUTE(order_amount * exchange_rate),

Bidirectional (Active-Active) Replication

In bidirectional replication, both databases accept writes simultaneously and changes flow in both directions. It's the most powerful — and most complex — GoldenGate configuration.

DATABASE A (Primary DC) DATABASE B (Secondary DC) Extract A (extA) ─────────────────────▶ Replicat B (repB) Replicat A (repA) ◀───────────────────── Extract B (extB) Both databases: READ + WRITE (active-active) Loop prevention: CRITICAL — without it, changes loop infinitely

Loop Prevention (Critical)

Without loop prevention, a change made on Database A gets replicated to B, then B's Extract captures it and sends it back to A, which sends it to B again — an infinite loop. GoldenGate prevents this using transaction tags.

-- On Database A's Extract (extA.prm):
EXTRACT extA
TRANLOGOPTIONS EXCLUDETAG 00    ← Exclude changes tagged with '00' (Replicat's tag)
EXTTRAIL ./dirdat/ea
TABLE hr.*;

-- Replicat automatically stamps its applied changes with tag 00.
-- Extract sees that tag and skips those records — loop broken!
✅
How it works When Replicat applies a transaction, Oracle writes a special marker (tag 00) in the redo log. The Extract on that same database reads the redo log and skips anything tagged 00 — because that came from the other database, not from a local application write.

Conflict Detection & Resolution (CDR)

When the same row is modified on both databases at the same time, you have a conflict. GoldenGate's CDR policies handle this automatically.

-- In repA.prm (Database A's Replicat):
MAP hr.employees, TARGET hr.employees,
  RESOLVECONFLICT (UPDATEROWEXISTS,
    (DEFAULT, USEMAX (last_updated)));    ← Most-recent timestamp wins

-- Ignore delete conflicts (row already gone):
MAP hr.orders, TARGET hr.orders,
  RESOLVECONFLICT (DELETEROWEXISTS, (DEFAULT, DISCARD));

-- Use numeric delta for counters (add the difference, not overwrite):
MAP hr.inventory, TARGET hr.inventory,
  RESOLVECONFLICT (UPDATEROWEXISTS,
    (DEFAULT, USEDELTA (quantity_on_hand)));

CDR Resolution Strategies

StrategyDescriptionBest For
USEMAXKeep the larger value of the conflicting columnVersion numbers, timestamps (most-recent-wins)
USEMINKeep the smaller valuePriority queues, minimum bids
USEDELTAApply the numeric delta (difference) from sourceAccount balances, inventory counts
USELARGESTKeep the entire row from whichever side had the larger specified column valueMost-recent-row-wins scenarios
OVERWRITESource always overwrites targetHA failover where one side is master
DISCARDIgnore the conflict, keep target valueDelete-row-not-found conflicts

GoldenGate Security

Securing GoldenGate involves encrypting database passwords in parameter files, encrypting trail files, and securing the network transport. Never store plain-text passwords.

Password Encryption (Basic)

-- Encrypt a password in GGSCI:
GGSCI> ENCRYPT PASSWORD Oracle_4U BLOWFISH
-- Output: AACAAAAAAAAAAAJAUZCDCDABFAAAAAAA

-- Use it in parameter files:
USERID ggadmin, PASSWORD AACAAAAAAAAAAAJAUZCDCDABFAAAAAAA ENCRYPTKEY DEFAULT

Oracle Wallet / Credential Store (Recommended)

-- Create the credential store:
GGSCI> CREATE CREDENTIALSTORE

-- Add a credential alias:
GGSCI> ALTER CREDENTIALSTORE ADD USER ggadmin@orcl PASSWORD Oracle_4U ALIAS ggalias

-- Now in ALL parameter files, replace USERID/PASSWORD with:
USERIDALIAS ggalias         ← No password visible anywhere

-- Verify:
GGSCI> INFO CREDENTIALSTORE

Trail File Encryption

-- Generate a master key:
GGSCI> CREATE MASTERKEY
GGSCI> ENCRYPT MASTERKEY AES256

-- In Extract parameter file:
ENCRYPTTRAIL AES256

-- In Replicat parameter file:
DECRYPTTRAIL AES256

Network Encryption (Microservices)

# In oggca.rsp for MA deployments:
SSL_ENABLED=true
SSL_CERTIFICATE=/path/to/cert.pem
SSL_PRIVATE_KEY=/path/to/key.pem
SSL_CA_CERTIFICATE=/path/to/ca.pem
✅
Security best practices summary (1) Always use credential store aliases, never plain-text passwords. (2) Encrypt trail files with AES-256 in any environment that's not fully isolated. (3) Use TLS for MA deployments. (4) Restrict OS access to $OGG_HOME — trail files contain your production data.

BUFSIZE 64

EXTTRAIL ./dirdat/ex

-- $OGG_HOME/dirprm/rephr.prm — High-throughput settings:
REPLICAT rephr
USERIDALIAS ggalias
ASSUMETARGETDEFS

-- Parallel apply (most impactful tuning option):
PARALLELISM 4                   ← Start with 4 threads
MAXPARALLELISM 8                ← Allow auto-scaling up to 8

-- Batch SQL for bulk efficiency:
BATCHSQL
BATCHSQLOPTIONS OPBUFSIZE 4096, MAXBATCHSIZE 1024

-- Group transactions to reduce commit overhead:
GROUPTRANSOPS 1000              ← Group up to 1000 ops per commit

MAP hr.*,  TARGET hr.*;

Troubleshooting Guide

A systematic approach to diagnosing and resolving GoldenGate issues — from minor ABENDs to major re-sync scenarios.

Troubleshooting Framework (Always Follow This Order)

  1. Run INFO ALL — identify which process is ABENDED or STOPPED
  2. Run VIEW REPORT <processname> — find the error at the bottom
  3. Run VIEW GGSEVT — check for system-level events
  4. Check OS logs: /var/log/messages, Oracle alert log
  5. Use LOGDUMP to inspect trail file contents if needed
  6. Check database views: V$GOLDENGATE_STATS, alert log

Common Errors & Resolutions

ErrorRoot CauseResolution
OGG-00446 — Error opening log fileArchive log deleted before Extract captured itIncrease archive retention; check ARCHIVELOG_DEST
OGG-01161 — Bad column lengthColumn definition mismatch source vs. targetRun DEFGEN; update SOURCEDEFS; check column sizes
ORA-01403 — No data foundDelete/update for a row that doesn't exist on targetAdd REPERROR(1403, DISCARD); verify initial load completed
ORA-00001 — Unique constraintDuplicate key — initial load + Replicat overlapUse HANDLECOLLISIONS during sync phase; remove after
OGG-01224 — TCP/IP errorNetwork connectivity to target Manager failedCheck firewall; verify Manager running; verify port 7809
OGG-00664 — OCI error fetching rowLong-running uncommitted transaction holding read positionFind and commit/rollback the long transaction on source
Extract ABEND: missing archived logArchives purged before Extract processed themIncrease archive retention; consider ARCH_LOG_LATE_ARRIVAL

Emergency Re-Synchronization Procedure

  1. Stop all GoldenGate processes immediately
  2. Document the situation: what trail files exist, last known good Replicat position
  3. If trail files cover the gap → reset Replicat to last good checkpoint and restart
  4. If trail files are missing or data is corrupt → perform new initial load (see Module 6)
  5. After re-sync: start with HANDLECOLLISIONS enabled, monitor closely for 24 hours, then remove it
-- Check Replicat's last checkpoint (last successfully applied SCN):
SELECT group_name, log_csn, log_position
FROM   ggadmin.chkpt
ORDER BY last_update_ts DESC FETCH FIRST 5 ROWS ONLY;

  'GGG Param',       VALUE                         FROM V$PARAMETER WHERE NAME='enable_goldengate_replication'
UNION ALL SELECT
  'Suppl. Logging',
  'MIN:' || SUPPLEMENTAL_LOG_DATA_MIN ||
  ' PK:' || SUPPLEMENTAL_LOG_DATA_PK  ||
  ' ALL:' || SUPPLEMENTAL_LOG_DATA_ALL  FROM V$DATABASE;

-- Check active archive destinations:
SELECT DEST_NAME, STATUS, TARGET, ARCHIVER
FROM  V$ARCHIVE_DEST
WHERE STATUS = 'VALID';

-- Check GoldenGate statistics (if using Integrated Extract):
SELECT * FROM V$GOLDENGATE_STATS;

-- Check for long-running uncommitted transactions (high lag cause):
SELECT s.username, t.start_time, t.used_ublk, s.status
FROM  v$transaction t JOIN v$session s ON t.ses_addr = s.saddr
ORDER BY t.start_time;
